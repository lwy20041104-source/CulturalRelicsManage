## 一、技术相关问题

### 1.1 架构设计

**Q1：为什么采用前后端分离架构？相比传统单体架构有什么优势？**

**答**：前后端分离架构将前端（Vue 3）与后端（Spring Boot）完全解耦，通过 RESTful API 进行通信，具备以下优势：

- **职责分离**：前端专注 UI 与交互体验，后端专注业务逻辑与数据处理
- **独立部署**：前后端可独立构建、测试、部署，互不干扰
- **技术异构**：前后端可各自选择最适合的技术栈（Vue 3 + Element Plus / Spring Boot + MyBatis）
- **可扩展性**：后端 API 可同时服务 Web、移动端（微信小程序）、第三方系统
- **团队协作**：前后端开发人员可并行开发，提高开发效率

**Q2：系统采用的三层架构（Controller-Service-Mapper）是如何分工的？**

**答**：

| 层次 | 职责 | 示例 |
|------|------|------|
| **Controller 层** | 接收 HTTP 请求，参数校验，调用 Service，返回响应 | `CulturalRelicController` 处理文物 CRUD 请求 |
| **Service 层** | 业务逻辑处理，事务管理（`@Transactional`），数据转换 | `CulturalRelicServiceImpl` 处理文物编号自动生成 |
| **Mapper 层** | 数据库操作，SQL 映射（MyBatis XML），与数据库交互 | `CulturalRelicMapper` 执行 SQL 查询 |

**依赖方向**：Controller → Service 接口（依赖倒置）→ Mapper。Service 层通过**接口抽象**实现，便于单元测试 Mock 和未来扩展。

**Q3：系统如何保证安全性？**

**答**：本系统实现了多层安全防护：

- **认证层**：JWT Token 无状态认证，Token 有效期 2 小时，可刷新
- **授权层**：Spring Security + RBAC（4 种角色：ADMIN / CURATOR / APPROVER / LOANER）
- **密码安全**：BCrypt 加密（10 轮加盐迭代）
- **登录安全**：5 次失败锁定 30 分钟，IP 地址记录与审计
- **SQL 注入防护**：100% 使用 MyBatis 预编译语句（`#{}`）
- **XSS 防护**：输入校验与转义
- **敏感信息脱敏**：密码、手机号、邮箱在日志中自动脱敏
- **CORS 配置**：精确控制跨域访问来源

### 1.2 后端技术

**Q4：为什么选择 MyBatis 而非 JPA/Hibernate？**

**答**：

| 对比维度 | MyBatis | JPA/Hibernate |
|----------|---------|---------------|
| SQL 控制 | 开发者完全控制 SQL，便于优化 | SQL 自动生成，复杂查询难以优化 |
| 学习曲线 | 低，SQL 基础即可上手 | 高，需理解 JPA 规范与实体生命周期 |
| 复杂查询 | XML 动态 SQL，灵活应对多条件筛选 | JPQL/Criteria API，复杂场景代码冗长 |
| 本系统需求 | 文物多条件筛选、分类树递归查询、统计报表 | — |

本系统存在大量复杂动态查询（如文物多条件组合筛选、借展状态联合查询），MyBatis 的动态 SQL（`<if>`、`<foreach>`）能更灵活高效地处理这些场景。

**Q5：项目中如何使用 Redis？**

**答**：Redis 在本系统中承担以下角色：

- **JWT Token 黑名单**：用户登出后 Token 加入 Redis 黑名单，防止重用
- **登录安全**：记录登录失败次数（5 次限制）、IP 锁定状态（30 分钟）
- **缓存**：文物分类树、统计数据、热点查询结果（通过 `CacheService` 管理）
- **会话管理**：支持分布式部署下的用户状态共享

使用 Spring Data Redis + Jackson 序列化，配置了自定义 `RedisTemplate`，Key 使用统一前缀，通过 `CacheConstants` 管理缓存 Key 规范。

**Q6：数据库设计中如何保证数据一致性和完整性？**

**答**：

- **事务管理**：Service 层关键操作使用 `@Transactional` 注解，保证原子性
- **外键约束**：核心表之间建立外键关联（如文物与分类、用户与博物馆）
- **业务规则验证**：前后端双重验证（如：不能重复申请修复中的文物、借展日期必须 ≥ 今天）
- **状态机模式**：借展和修复使用严格的状态流转（如：待审批 → 借出中 → 已归还，不允许跳跃）
- **编号唯一性**：文物编号（WW00001）、专家编号（EXP00001）、档案编号（AR-年份-序号）自动生成且唯一

### 1.3 前端技术

**Q7：为什么选择 Vue 3 + Element Plus 作为前端技术栈？**

**答**：

- **Vue 3 Composition API**：相比 Options API，Composition API 提供更好的逻辑复用和代码组织能力，适合大型项目
- **Element Plus**：成熟的 Vue 3 UI 组件库，提供 Table、Form、Dialog、Tree 等丰富组件，大幅减少 UI 开发工作量
- **Pinia**：Vue 3 官方推荐的轻量级状态管理方案，比 Vuex 更简洁、类型安全
- **Vite**：极快的冷启动和热更新（HMR），开发体验显著优于 Webpack
- **TypeScript 支持**：Vue 3 + Pinia + Element Plus 均有良好的 TypeScript 类型推导

**Q8：前端如何处理 WebSocket 实时通知？**

**答**：

- 使用 **SockJS** + **STOMP** 协议建立 WebSocket 连接
- 服务端通过 `@SendToUser` 注解实现**用户私有队列**，精准推送通知
- 前端 `NotificationBell.vue` 组件订阅个人通知频道，实时更新未读数
- 通知类型包括：借展申请、审批结果、逾期提醒、修复申请等
- 支持**离线消息**：通知持久化到数据库，用户登录后可查看历史通知

**Q9：主题系统（6 个主题 × 2 种模式 = 12 种风格）是如何实现的？**

**答**：

- **全局 CSS 变量系统**：定义主题色、背景色、文字色等 CSS Custom Properties
- **主题配置对象**：6 个预设主题（古典棕色、青瓷雅韵、青花瓷韵、紫檀沉香、墨玉清幽、琥珀金辉）各有独立的颜色方案
- **暗黑模式**：在每个主题基础上叠加暗色变量覆盖，明暗切换时修改 CSS 变量值
- **持久化**：用户选择存入 `localStorage`，刷新页面后自动恢复
- **过渡动画**：所有颜色变化使用 `transition: 0.3s` 平滑过渡
- **可访问性**：严格遵循 WCAG 2.1 对比度标准，确保护眼和可读性

### 创新点一：3D 文物展示与交互

**Q16：3D 文物展示是如何实现的？**

**答**：

**技术方案**：基于 **Three.js 0.160.1** 构建 3D 渲染引擎

- **渲染器**：`WebGLRenderer`（硬件加速）
- **相机**：`PerspectiveCamera`（透视投影，真实感强）
- **控制器**：`OrbitControls`（鼠标旋转、缩放、平移）
- **模型加载器**：`GLTFLoader`（GLTF/GLB）、`OBJLoader`（OBJ）

**光照系统**：
- 环境光（`AmbientLight`）：模拟全局环境照明
- 方向光（`DirectionalLight`）：模拟定向光源，产生明暗对比
- 可调节光照强度和背景颜色

**全流程支持**：
- **上传**：拖拽/点击上传，文件类型验证，50MB 限制
- **存储**：文件存储 + 数据库记录（URL、类型、大小、上传时间）
- **查看**：详情对话框"查看 3D 模型"按钮，列表"3D 模型"列图标
- **交互**：自动旋转、重置视角、全屏显示、网格切换

**技术难点**：
- 不同格式模型的坐标系统一（GLTF 右手系 vs OBJ 的 Y-up 调整）
- 模型自动缩放与居中算法（计算包围盒，适配视口）
- 大文件（50MB）加载优化（显示加载进度）

**Q17：3D 模型管理涉及哪些后端 API？**

**答**：

| 接口 | 方法 | 功能 |
|------|------|------|
| `/api/relics/{id}/3d-model` | POST | 上传/替换 3D 模型 |
| `/api/relics/{id}/3d-model` | DELETE | 删除 3D 模型 |
| `/api/relics/{id}/3d-model` | GET | 获取 3D 模型信息 |

关键实现细节：
- 文件存储在 `uploads/3d-models/` 目录下
- 删除时同时清理物理文件和数据库记录
- 使用自定义 `clear3DModelInfo()` SQL 显式设置 NULL（解决 MyBatis 动态 SQL 不处理 NULL 值的问题）

### 创新点二：AI 智能查询

**Q18：AI 智能查询的"双层检索策略"是什么？**

**答**：

**第一层：馆藏优先**
- 用户输入查询关键词（如"青花瓷"）
- 系统首先在本地文物数据库中进行**关键词匹配 + 同义词扩展**
- 同义词扩展示例："青花瓷" → ["青花", "青花瓷器", "元青花", "明青花"]
- 对匹配结果进行**相关度评分**（0-100 分）

**第二层：全网兜底**
- 当本地馆藏无匹配结果时，自动触发**全网检索**
- 通过 HTTP 爬虫搜索互联网文物信息
- **7 层图片提取策略**：从 HTML `<img>` → CSS background → Open Graph → JSON-LD → Schema.org → 备用图库 → 默认图片
- 返回结果包含标题、描述、图片、来源链接

**核心优势**：
- 确保用户**总能获得结果**（本地无 → 全网有）
- 本地结果优先，保证准确性和响应速度
- 全网检索作为补充，扩大知识覆盖范围

**Q19：AI 查询的会话管理是如何设计的？**

**答**：

- 每个用户可创建多个**查询会话**（`ai_chat_session` 表）
- 会话内支持**多轮对话**，上下文连续
- 查询结果**持久化存储**（`ai_query_result` 表），支持历史回溯
- 消息记录完整保存（`ai_chat_message` 表），支持会话恢复
- 通过 `DeepSeekService` 集成大语言模型，提供自然语言理解能力

### 创新点三：实时通知系统

**Q20：实时通知系统如何实现精准推送？**

**答**：

**技术架构**：
- **WebSocket + STOMP 协议**：建立持久化双向通信连接

5. 系统公告 → 全体用户

**定时任务**：
- 使用 `@Scheduled` 注解，每天凌晨自动检查逾期借展
- 自动向逾期用户推送提醒通知

**前端实现**：
- `NotificationBell.vue` 组件实时更新未读数
- `SockJS + @stomp/stompjs` 客户端库处理 WebSocket 连接
- 通知持久化到数据库，支持离线消息

## 四、综合问题

**Q23：项目开发过程中遇到的最大技术难点是什么？是如何解决的？**

**答**：以下是几个关键技术难点及解决方案：

**难点一：3D 模型删除时 MyBatis 不更新 NULL 值**
- **问题**：MyBatis `updateById()` 的动态 SQL 跳过 NULL 字段，导致删除 3D 模型链接失败
- **解决**：编写自定义 SQL `clear3DModelInfo()`，显式 SET 字段为 NULL

**难点二：3D 模型详情页返回后分页状态丢失**
- **问题**：`router.back()` 无法保留列表页的页码和搜索状态
- **解决**：通过 URL 查询参数传递状态（页码、搜索条件），返回时恢复

**难点三：AES 加密备份密钥长度错误**
- **问题**：Java 安全策略限制 256 位加密
- **解决**：改用 128 位 AES 加密，使用 16 字节固定密钥

**难点四：文物多条件动态查询的性能优化**
- **问题**：多字段组合筛选导致 SQL 复杂
- **解决**：利用 MyBatis 动态 SQL（`<if>`、`<where>`）按需拼接查询条件，配合数据库索引优化
